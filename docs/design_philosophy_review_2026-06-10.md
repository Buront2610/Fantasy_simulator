# 設計思想レビュー: 思想の判別・現状の体現度・再設計対象（2026-06-10）

リポジトリのコード・ドキュメント・CI構成から設計思想を判別し、各原則の体現度と意図との乖離を評価し、
再設計対象を優先度付きで整理したもの。

- 関連: `docs/review_report_2026-06-10.md`（総合レビュー。バグ・バランス・ロードマップはそちら）
- オーナーが意識していると申告した原則: DDD / クリーンアーキテクチャ / DbC / TDD / SRP / 状態機械
- 判別結果: **申告された原則はすべてコードから検出可能**。加えて申告にない CQRS／イベント台帳
  ハイブリッドと、フィットネス関数文化が存在する

## 1. 設計思想のまとめ（検出された思想の全体像）

| 原則 | 体現度 | 一言評価 |
|---|---|---|
| DDD（戦術パターン） | ◎ | `world_change/` は値オブジェクト・コマンド・仕様・リデューサの見本市 |
| DDD（戦略パターン） | △ | コンテキストマップ文書はあるが、最大コンテキストが物理化されていない |
| クリーンアーキテクチャ | ◎ | 依存方向をフィットネス関数で CI 強制。稀有な水準 |
| CQRS / イベント台帳（暗黙） | ○ | 明確に存在するが、ハイブリッドの線引きが暗黙的（季節デルタバグの根） |
| DbC | ○ | fail-fast と clamp-and-continue の二流派が共存。規約未明文化 |
| TDD | ◎ | テスト:コード比 0.99:1。空白は創発的性質のみ（総合レポート §15 で対処） |
| SRP | ◎（過剰） | ファイル単位は完璧。代償としてパッケージ凝集が失われている |
| 状態機械 | ◎ | 明示的・遷移検証付き・ADR で文書化 |

### 各原則の証拠と乖離

#### DDD

- **証拠（戦略）**: `docs/domain/context_map.md`、`docs/domain/ubiquitous_language.md` —
  コンテキストマップとユビキタス言語の文書化は戦略的 DDD の教科書的アーティファクト
- **証拠（戦術）**: `world_change/` の commands（8種）→ specifications → changesets → reducers →
  domain_events の流れ。`FactionId` 等の値オブジェクト、`validate_declare_war_command` の
  コマンド検証。`world_*_api.py` / `_facade.py` のアグリゲート・ファサード、
  `WorldEventRecorderPort`（`world_event/api.py:43`）の明示的ポート
- **乖離**: 境界づけられたコンテキストが物理化されているのは新しい部分のみ
  （`world_change/` `observation/` `language/`）。最古かつ最大の「world」コンテキストは
  ルート直下 103 モジュールのフラット名前空間で、コンテキストが命名規則（`world_*`）にしか
  存在しない。また `World` は全ドメイン（キャラ・地形・噂・言語・暦・イベント）がぶら下がる
  **ゴッド・アグリゲート**（ファサード分割で緩和済みだが集約境界としては未設計）

#### クリーンアーキテクチャ

- **証拠**: `scripts/architecture_guard.py`（AST 解析、1,060 行）＋ `architecture_guard.json` ＋
  `tests/test_architecture_guard.py` の CI 強制。simulation→ui/persistence 禁止、ヘッドレス層の
  print/input/open 禁止、決定論層の random/datetime import 禁止。`removal_condition` 付き
  override 32 件の負債台帳。`InputBackend` / `RenderBackend` / `Combatant` 等の Protocol が
  ポート＆アダプタの語彙でドメイン・UI 両層に分布
- **乖離**: 既知の穴 — `world.py:265` の ui 遅延 import、`character_creator/interactive.py` の
  ui 直 import、ルートドメイン層→ui の禁止ルール不在。ミックスインの暗黙的 self 共有
  （カプセル化の弱点。外部設計書 docx も同一指摘）

#### CQRS / イベント台帳ハイブリッド（申告外・検出）

- **証拠**: 書き込み系 = command → reducer → canonical `WorldEventRecord`。
  読み取り系 = `observation/` の projection（war_map / location_history / route_status /
  world_change_report）。`simulation/queries.py`、UI の view_models
- **性格**: 完全なイベントソーシングではなく**「可変状態＋イベント台帳」のハイブリッド**。
  状態はイベントから再構築されない。ADR-0001 / ADR-0003 がこの境界を扱っている
- **乖離（重要）**: 「どの状態がイベント由来で、どの状態が独立して保存されるか」の線引きが
  暗黙的。**季節デルタバグ（総合レポート §3 #3）はこのハイブリッドの典型的な失敗モード** —
  状態がイベントから導出可能なら、シリアライズ漏れによる焼き付きは構造的に起き得なかった

#### DbC（Design by Contract）

- **証拠**: `WorldEventRecord.__post_init__` の契約検証、ロード時トポロジー検証
  （重複 route ID・自己ループで即 ValueError）、record_id 重複の fail-fast、
  `Combatant` 等の Protocol による構造契約
- **乖離**: 二流派の無自覚な共存 —
  **fail-fast 契約**（識別子・永続化境界: 違反を拒否）と
  **clamp-and-continue**（ゲームプレイ値: ステータス [1,100]、関係値 [±100] — 違反を黙って矯正）。
  使い分け自体は合理的（データ整合性は止める、ゲーム数値は進める）だが規約として未明文化のため、
  AI エージェントがどちらの流儀で新コードを書くべきか判断できない

#### TDD

- **証拠**: テスト約 36,500 行 / コード約 36,900 行（比 0.99:1）、101 テストファイル。
  全マイグレーションに現実的 fixture、破損系（不正 JSON・重複 ID・改竄 rng_state）の網羅、
  アーキテクチャガード自体がテストとして実行される
- **限界**: 成果物からテストファーストかテスト随伴かは判別不能。空白地帯は正確に
  「実行時の創発的性質」（バランス・長期挙動 — 人口絶滅・結婚 0 件は全テストグリーンのまま発生）。
  総合レポート §15 World Health Check が対処

#### SRP

- **証拠**: 複雑度バジェット（cyclomatic 20 / 関数 80 行 / クラス 220 行 / 公開メソッド 12）の
  機械的強制。`Simulator` は 8 ミックスイン・112 行。`simulator.py` は 11 行の純粋再エクスポート。
  平均ファイルサイズ約 160 行
- **乖離**: **遵守が過剰側に振れている**。「1 ファイル = 1 責務」は達成したが
  「1 パッケージ = 1 コンテキスト」が未達。ルート 103 モジュール（world_* だけで 45）の
  フラット名前空間は SRP の代償としての凝集喪失
- **ゲート較正の注意**: 既存の complexity / size gate が、意味のある凝集よりも機械的分割を促している
  可能性もある。SRP 改善は「ファイルを小さくする」ではなく、変更理由・所有境界・読み手の移動量を
  減らすことを基準にし、閾値強化や `max_module_lines` 追加は過剰規制にならないかを先に監査する。

#### 状態機械

- **証拠**: `adventure_state_machine.py`、`world_change/state_machines.py`（遷移検証関数）、
  injury ladder（none→injured→serious→dying）、`timeline_pipeline.py` の宣言的フェーズ順序、
  ADR-0002「state machine boundaries」
- **乖離**: 特になし。今後の WorldArc（総合レポート §6）もこのパターンの延長で実装できる

### 申告外に存在する文化（資産として認識すべきもの)

- **フィットネス関数文化**: アーキテクチャ制約を「文書」でなく「落ちるテスト」で表現する
- **負債台帳文化**: `removal_condition` 付き override、xfail 台帳（§15 提案）と同型
- **ADR 文化**: `docs/adr/` 5 件。意思決定の記録が機能している
- **AI エージェント協働文化**: `docs/agent_roles/`、`session_handoffs/`、`subagent_contract.md`、
  CLAUDE.md / AGENTS.md — マルチ AI 開発を前提としたドキュメント構造

## 2. 現状の総括

意図と実装の一致度は全体として非常に高い。乖離はほぼすべて
**「最も古い層（ルートの world コンテキスト）に、新しい規律が遡及適用されていない」**
という一点に収束する。新しいパッケージ（world_change / observation / language）は
申告された全原則を高水準で体現しており、設計思想は「これから書くコード」に対しては
正しく機能している。問題は遺産部分との二重構造であり、思想の欠陥ではない。

## 5. 思想そのものの妥当性判定

| 問い | 答え |
|---|---|
| 思想は正しいか | **正しい**。AI 主導開発 × シミュレーションゲームという条件で費用対効果が成立している |
| 完全か | 否。「守り」6 原則に対し「体験を計測で守る」原則が欠けていた |
| リスクは | 設計過多（docx リスク表と同診断）。YAGNI を明示原則に昇格させて対抗する |

### 5.1 正しいと判定する根拠

1. **AI エージェント主体の開発体制が費用対効果を逆転させている**。人間のソロ開発なら
   オーバーエンジニアリングだが、AI が大半のコードを書く体制では、機械が強制できる制約
   （ガード・契約・テスト・ADR）だけがセッションをまたいで持続する「記憶と判断力」になる。
   証拠はこのリポジトリ自体 — AI 主導で 7.3 万行まで育って一貫性を保っている
2. **ジャンルと思想が一致している**。決定論ルール＝セーブ/リプレイ再現性、状態機械＝
   冒険/戦争/負傷の進行、イベント台帳＋projection＝DF Legends のアーキテクチャそのもの。
   エンタープライズ思想のカーゴカルトではない
3. **今後の全設計（WorldArc / TRPG 戦闘 / AppService / Personal Mode）が既存の語彙で
   表現できた**。思想が間違っていれば新機能設計のたびに思想と戦うことになるが、そうなっていない

### 5.2 欠けていたもの

申告された 6 原則はすべて「守り」の思想（正しさ・変更容易性・構造）であり、
**「世界が面白いか・生きているか」を守る思想がセットに入っていなかった**。
結婚 0 件・人口絶滅は 6 原則のどれにも違反せず発生した（無瑕の構造の上の死んだ世界）。
World Health Check（総合レポート §15）はバグ対策である以上に、思想セットの欠落を埋める補完である。

### 5.3 設計規約（原則の昇格・明文化）

以下を正式な開発規約とし、CLAUDE.md / AGENTS.md に運用形で反映する
（AI エージェントが確実に読む場所はそこだけであるため）。

**規約 7: 構造はテストで守る。体験は計測で守る**
バランス・確率・進行速度に影響する変更は、統計的検証（World Health Check のヘルスバンド、
導入までは playtest スクリプト）を通す。バンドの変更は「テストを黙らせる」行為ではなく
「設計判断の記録」であり、同一 PR で理由とともに行う。

**規約 8: YAGNI — 抽象は 2 例目の具体が要求するまで作らない**
汎用エンジン・基盤・フレームワークを先に作らない。1 例目は具体的に書き切り（例: 戦争アーク）、
2 例目（例: 疫病アーク）を書くときに共通部分を抽出する。この思想セットの既知の失敗モードは
設計過多であり、本規約がその対抗原理である。

**規約 9: 契約の使い分け（DbC 二流派の明文化）**（再設計対象 R3）
- 識別子・参照・永続化境界の不正 = **fail-fast**（例外で拒否。例: record_id 重複、トポロジー不整合）
- ゲームプレイ数値（ステータス・関係値等）= **clamp-and-continue**（黙って矯正し進行を続ける）
新コードはこの使い分けに従う。迷ったら「壊れたデータが保存されうるか？」で判定し、
保存されうるなら fail-fast 側に倒す。

**規約 10: プロセス状態は必ずシリアライズ**（再設計対象 R1）
複数ティックにまたがる進行中状態（補正の帳簿・アーク・保留選択・蓄積圧力）は、
導入初日から to_dict/from_dict に含める。一時補正は「保存値の書き換え＋後で巻き戻し」ではなく
「読み出し時に合成する派生値」を優先する（季節デルタバグの教訓）。
