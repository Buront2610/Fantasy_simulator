## 1. 総評

コードベースの「守り」は非常に堅牢。年率→日率の確率変換の数学的一貫性、防御的クランプの徹底、
アーキテクチャ・フィットネス・ガード（AST 解析＋負債台帳＋CI 強制）、テスト比ほぼ 1:1、
i18n の TEXT 776 キー完全パリティは、この規模のプロジェクトとして例外的な水準。

一方で「攻め」＝歴史生成機としての面白さの側に構造的な問題が集中している。
最大の発見はテストプレイで判明した **「世界の必然的な死」**（人口補充メカニクス不在）と、
**「世界変化が因果のないサイコロであること」**（戦争が台帳上にしか存在しない）。

## 2. テストプレイ実測サマリ

設定: 初期人口 20、100 年、シード 7 / 42 / 1234。世界変化計測時は `world_changes_per_year=1`（UI の実設定値）。

| 計測項目 | 結果 |
|---|---|
| 人口推移 | 20 → 100年後 2〜7 人に単調減少（出生メカニクス不在） |
| 結婚 | **200 シミュレーション年で成婚 0 件** |
| 関係値 | 実測平均 -4〜-24（正に飽和せず、battle の -20/-30 が meeting の期待値 +5 を打ち消す） |
| 長寿種族の収束 | 生存エルダー全員が **wisdom=100 / dexterity=1〜2** の同一プロファイル |
| スキル | 平均 8.2/10 まで成長するが、**判定に使う箇所がゼロ**（完全な死蔵） |
| 冒険結果 | safe_return 60% / injury 30% / **death 0%**（86 冒険） |
| イベント構成 | 総レコード約 2,200 件のうち **49% が aging** |
| 時代（era） | **100 年で 16 回シフト**（平均 6 年に 1 回、連年・年 2 回もあり） |
| 戦争 | 宣言 9 / 終結 7、期間 0〜16 年。**中身は占領 6〜9 件のみ、会戦ゼロ、死者ゼロ** |

### バランス上の問題（バグではないが要対応）

- **冒険リスクの飽和**: 負傷率クランプ `[0.04, 0.22]`（`adventure_policy.py:112-125`）に対し、平均パーティ × danger 80 で既に 0.234 → 上限張り付き。danger 15〜80 の差もポリシー差（assault ×1.30）も結果に反映されない。戦利品率は cautious でも 0.61、treasure 政策で 0.95 飽和。→ クランプを `[0.02, 0.45]` 程度へ拡幅、danger を loot 側にも反映
- **加齢帯が絶対年齢固定**（`events/lifecycle.py:33-67`）: エルフ（寿命600）は中年帯を400年過ごし wis100/dex1 へ収束。オーク（寿命60）は中年帯が存在しない。→ `age / max_age` 比率ベースへ
- **結婚閾値が到達不能**（`events/relationships.py:67`）: rel≥60×2 かつ avg≥70 に対し関係値実測平均 -7。同一ペアへのイベント反復集中もないため 200 年で 0 件
- **イベント予算が人口非依存**: 世界全体で年 8 件固定（`timeline.py:60-67`）、冒険発生も年率 25% 固定。人口を増やすほど一人当たりの歴史が薄まる。→ per-capita 化
- **負け犬スパイラル**（連敗→con 流出→死）は歴史生成機としてはむしろ資産。「全敗者が同じ末路を辿る」単調さのみ要改善

## 4. ゲームデザイン診断 — 歴史生成機としての「アカン」ランキング

ビジョン（DF Legends / 探索遥かなり）の物差しで致命度順:

1. **世界が一代で終わる** — 出生なし＋結婚 0 件で人口単調減少。歴史の根幹である「世代」（王朝・血讐・継承）が構造的に生成不能。生成される歴史は「20 人が順番に死ぬ話」一種類のみ
2. **キャラクターが収束する** — 全長寿種族が同一プロファイルへ、スキルは無接続。歴史記録の主語が交換可能になり「あの○○がやった」という固有性が出ない。過去が未来の確率を変えない
3. **悲劇が起きない** — 冒険 death 0%。「全滅した遠征」「一人だけの生還者」が出ない世界の歴史は読む価値が薄い
4. **戦闘ログが嘘** — 勝敗は 1 回の `power + randint(0,30)` 比較、3 ラウンドのログは結果から後付け逆算した固定脚本（`combat.py:97-110`）
5. **歴史の密度が人口に比例しない** — イベント予算が世界全体固定
6. **社会構造が 1 対 1 ペアのみ** — 血縁・家系・派閥所属がなく、関係のネットワークが育たない

## 5. 世界変化の診断 —「目に見える変化が少ない」の原因

体感は実測で裏付けられた。原因はインフラ不足ではなく、**生成ポリシー（`world_change_driver.py`、389 行）が世界を物語にしていない**こと。

### 構造的問題: すべてが因果のないサイコロ

`generate_world_change`（`world_change_driver.py:371-388`）は 7 種のジェネレータから一様抽選し、各ジェネレータも世界状態をほぼ読まない:

- **戦争**: `build_war_declaration_change_set`（`changesets.py:172-205`）は**イベントレコード 1 件を積むだけ**。danger/safety/LocationState/キャラへの影響が一切ない。開戦はランダムな派閥ペア（authored の `faction_relationships` は地図投影専用で開戦判定に未使用）、戦場は派閥の地理と無関係なランダム 2 地点（`:63-70`）、終戦は抽選時 55% で原因・勝者なし
- **時代**: ランダムな別 era キーへ飛ぶだけ → 100 年で 16 回シフト
- **地形**: `_next_biome`（`:29-33`）はランダムな別バイオームへジャンプ
- **文明位相**: ランダム位相＋ランダム微小デルタの random walk

DF 的歴史の核心は「変化が前の状態から因果的に導かれる」こと。現状はポアソンノイズなので、プレイヤーが「流れ」を読み取れる構造が原理的に生まれない。

### 表示側の問題

履歴の 49% が aging。世界変化（100 年で約 80 件）はその海に埋没する。aging の severity を下げて
履歴ビューのデフォルトから間引くだけで体感密度は倍以上になる。

## 6. 今後の実装方針（議論の合意点）

### 6.1 方向性: 「イベントに変化を持たせる」＋「因果を持たせる」

「因果」は 3 部品に分解できる:

- **A. 進行中プロセスの永続状態（アーク）** — 「戦争 3 年目、西部戦線が膠着」という継続状態を持つエンティティ。長期シナリオの本体
- **B. 圧力の蓄積値** — 小イベントの累積が大変化を発火（派閥緊張→開戦、繁栄低下の累積→時代の終わり）
- **C. 因果リンク** — 出来事が「何のせいか」を指すポインタ

**C は配管済み**: `cause_key` / `cause_event_id` がドメインイベント→レコード `render_params` まで流れる
（`world_change/event_adapters.py:33,80`）。噂も `source_event_id` を持つ（`rumor/models.py:29`）。
ただし自然発生ドライバーは飾り文字列しか渡しておらず、**実イベント ID を一度も流していない**。

**A の実証済みパターンが AdventureRun**: 複数ステップ状態機械・日次予算進行・保留選択・outcome 付き解決・
シリアライズ対応。これを世界スケールへ一般化する。`world_change/state_machines.py` にも戦争遷移検証が既にある。

### 6.2 WorldArc（世界版 AdventureRun）の構想

```
WorldArc:
  arc_id, kind ("war" | "plague" | ...), phase（状態機械）
  participants（派閥 ID・地点 ID・関与キャラ ID）
  pressure / progress（蓄積値）
  origin_record_id（アークを始めたレコード = 因果の根）
  emitted_record_ids（このアークが生んだ出来事の系譜）
```

- 日次パイプラインに 1 フェーズ追加（`timeline_pipeline.py` の宣言的順序に乗せる）
- アーク発行レコードに `cause_event_id=origin_record_id` → 「講和←会戦←開戦←緊張」を遡れる系譜が自動的にでき、Legends 的「なぜ」ビューの土台になる
- ヘッドレス・決定論を守れば既存アーキテクチャガードの管轄にそのまま入る

**設計上の注意（合意済み）**:
1. **アーク状態は初日からシリアライズ対象**（スキーマ v9）。季節補正バグの教訓 — メモリ専用のプロセス状態は同型バグを再生産する
2. **汎用シナリオエンジンを先に作らない**。戦争を 1 本目のアークとして書き切り、2 本目（疫病・魔王・黄金時代等）で共通部分を抽出。1 本目の規約は「状態シリアライズ / cause_event_id / ヘッドレス」の 3 つのみ

### 6.3 計画中の機能への設計指針

**詳細戦闘ログ**
- 判定とログを同一乱数系列で生成（ラウンドごとの小ロールの積み上げで勝敗・負傷を決定）。現状の「結果から逆算した演出ログ」の上に詳細化を積まない
- ラウンドの命中/回避修正に**スキルを接続**（スキル死蔵とログの嘘を同時に解消）
- `(round, actor_id, action, roll, result)` の構造化レコードで保存し、文章化は既存の `summary_key` + `render_params` パターンで表示時に行う（i18n・セーブ容量・Legends 画面対応）
- 既存 `combat_log` 形式からの移行はスキーマ移行対象

**冒険者の自然増加（二系統）**
- **流入（移住・流れ者）= 人口の安定弁**: 発生率を地域の繁栄度（traffic/safety）に応答させる。即効性があり「世界の死」を止める
- **出生・世代 = 歴史の深さ**: 結婚閾値の再調整（または近接ペアへのイベント反復集中）が前提。parent/child タグ＋家名を同時導入（`add_relation_tag` 機構を流用）。成人年齢も `max_age` 比率で
- **随伴必須**: イベント予算・冒険発生率の per-capita 化と、地域収容力による人口上限（ないと指数増殖で日次パイプラインが破綻）

**戦争のプロセス化（WorldArc 1 本目）**
- 宣言時: 戦場地点の danger/safety に実効（`WorldScoreUpdate` 機構は changesets に既存）
- 進行中: 月次の戦争イベント（会戦・包囲・襲撃・難民）。戦場地点のキャラの battle 確率増・死亡経路開放 → 増えた冒険者が参加者/犠牲者になり、会戦が構造化戦闘ログの生成源になる
- 終戦: 占領数で勝者決定。征服/講和/共倒れの終わり方
- 開戦は faction tension（圧力値）駆動へ、時代シフトはアーク終結・圧力閾値の条件駆動へ（抽選から除外、100 年に 1〜2 回）

## 9. 追記: 冒険強化・サンドボックスモード・世界観の作り込みの方向性

オーナーの追加構想: (a) 冒険要素のさらなる強化、(b) 将来的に「この世界に一人の冒険者として乗り込む」
サンドボックス的モード、(c) その過程での世界観の作り込み（言語変化・文化変化。雛形あり）。

### 9.1 既存雛形の調査結果

- **言語エンジンは本格的な実装が既にある**: `fantasy_simulator/language/` 全 14 モジュール・約 1,840 行。
  音韻論（`phonology.py`）、音変化規則、系統継承（`parent_key`）、決定論的な `LanguageEvolutionRecord`、
  命名生成（`naming.py` 434 行）まで揃っている
- **シミュレーションにも接続済み**: 年次で `maybe_evolve_languages_for_year`（`world_language/__init__.py`）が
  呼ばれる（`world_calendar/api.py`）
- **ただしトリガーは `evolution_interval_years` の固定タイマー**。歴史と無関係に進化する —
  世界変化と同じ「因果のないサイコロ」問題（§5）がここにもある
- 文化系は静的データ（種族/部族/地域の命名規則、setting bundle の派閥・言語定義）として存在。動的変化は未実装

### 9.2 核心の認識: 「観察 → 選択的介入 → 憑依」の一本道

DF の Adventure モードの正体は「同じ世界シミュレーションの 1 エージェントを乗っ取る」こと。
乗っ取りに必要な部品はこのリポジトリに既に 3 つ揃っている:

1. **`AdventureRun.pending_choice` / `waiting_for_choice`**（`adventure_state_machine.py:36-77`）—
   「プレイヤーに聞くか、ポリシーが自動解決するか」の分岐点。サンドボックス＝この分岐を
   1 キャラについて常にプレイヤー側へ倒すこと
2. **UIContext / input_backend 抽象** — 対話的な意思決定ポイントの追加が安価
3. **お気に入りキャラの自動ポーズ**（`condition_worsened_favorite`）— 「特定キャラを追う」観察装置

**設計原則: 別ゲームモードを作らない。** プレイヤーキャラは日次パイプラインを流れる普通の `Character` で、
違いは「彼の意思決定だけ端末に聞きに来る」ことのみ。CLAUDE.md の「冒険はメインシミュレーション内に統合」を
サンドボックスにも適用する。エンジンを分岐させると、世界シミュ側への投資（戦争アーク・言語・文化）が
サンドボックスに乗らなくなる。時間粒度も日次パイプラインのまま、プレイヤーの選択点でだけ止まる
（リアルタイム化は不要、selective intervention の延長で成立）。

### 9.3 柱 1: 冒険強化はサンドボックスの先行投資として設計する

「観察しても面白く、操作しても面白い」形＝**冒険を世界の実資産に接続する**:

- **噂 → クエストフック**: 噂システム（信頼度・伝播・追跡フラグ）は事実上のクエストボード。
  冒険の行き先・目的を「その土地に流れている噂」から引く。観察では「噂が冒険者を動かした」歴史に、
  サンドボックスでは酒場で噂を聞いて行き先を選ぶ体験になる
- **ルート封鎖・戦争を冒険の障害に**: `route_blocked` 世界変化と戦場地点は存在するのに冒険の経路選択が
  読んでいない。封鎖迂回・戦場通過リスクで世界変化が冒険の手触りに直結する
- **Site の永続状態**: ダンジョンの「攻略済み度」を Site に持たせ、冒険の結果が場所に残る
- **名声/悪名**: 冒険結果を reputation に蓄積。観察では「英雄の台頭」、サンドボックスでは NPC 反応・
  パーティ勧誘の通貨。スキル接続と並ぶキャラ固有性の柱
- **選択ポイントの増設**: 撤退判断・ルート選択・戦利品の処遇など、ポリシーが自動解決できる選択点を
  増やすほど、後で「プレイヤーに聞く」へ切り替えられる面積が広がる

### 9.4 柱 2: 言語・文化は「歴史の投影」として作り込む

言語・文化を独立したタイマー駆動システムとして深掘りしない。WorldArc / cause_event_id（§6）をそのまま適用:

- **言語変化のトリガーを歴史イベントに**: 占領 → 支配派閥言語からの借用語、ルート長期封鎖 → 方言分岐
  （`parent_key` 系統機構を流用）、時代シフト → 命名流行の変化。`LanguageEvolutionRecord` は year 付き
  決定論的レコードなので、**生成タイミングを変えて `cause_event_id` を足すだけ**で「この音変化はあの戦争の
  せい」と遡れる言語史になる
- **可視化が肝**: 地名の経年変化（Millhaven → Milhaven）は「目に見える世界変化」の最良の形。
  world memory の alias 機構で旧名を噂・墓碑銘に残し、地図には現行名。サンドボックスでは
  「古い石碑だけ旧綴り」の体験になる
- **文化は小さく始める**: 地域/派閥ごとの `CultureProfile`（命名流行・祭事・タブー・職業重み程度）を
  WorldArc のイベントで漂移させる。「歴史の投影」パターンの 2 例目として、WorldArc 基盤抽出の材料にもなる

### 9.5 柱 3: 順序 — サンドボックスは世界が生きてから

テストプレイ実測（§2）がそのまま警告: 今サンドボックスを作ると、プレイヤーは「人口が減り続け、
結婚も戦争の実体もなく、時代だけが 6 年ごとに変わる世界」に降り立つ。一人の冒険者としての面白さは、
降り立つ世界の豊かさに完全に従属する。

§8 のロードマップと統合した推奨順:

1. 基盤修正（バグ #1/#4、クランプ拡幅、人口流入、per-capita 化）— 世界を生かす
2. 戦争アーク + 冒険の世界資産接続（噂フック、ルート/戦争連動、名声）— 遭遇するものを作る
3. **意思決定の「ポリシー界面」整備** — 冒険以外の選択（旅・訓練・関係）も徐々にポリシー経由へ。
   サンドボックスの技術的土台で、2 の作業中に少しずつ進められる
4. 言語・文化の歴史駆動化 — WorldArc に乗せ、地名漂移を地図と噂に出す
5. **サンドボックス MVP**: キャラ 1 人を作成または既存キャラに「憑依」し、お気に入り追跡＋全
   pending_choice をプレイヤーに回す。移動・滞在・パーティ参加の動詞を追加

「冒険強化」「世界観の作り込み」「サンドボックス」は別々の仕事ではなく、同じ一本の階段である。

## 11. 種族・魔法・スキルがほぼフレーバーである問題

実測・コード確認済みの現状:

- **スキル**: 判定に一切未接続（Lv10 と Lv0 が機械的に等価）。平均 8.2/10 まで育つのに何にも使われない
- **種族**: 作成時のステータス補正＋寿命のみ
- **魔法**: メカニクスとして存在しない。Fireball / Holy Light 等は TERMS の用語＝完全な飾り

これは §4 アカン #2「キャラクターが交換可能」の根本原因。対処方針は一貫して
**「専用システムを新造せず、既存の判定点に接続する」**:

- **スキル** → 戦闘ラウンドの命中/回避修正（§12）、冒険スコア、そして docx LG-6 の
  `CharacterLanguageKnowledge`（言語・Lore スキルで情報の表示解像度が変わる）が「スキルの
  ゲーム要素化」の最良の第一歩
- **魔法** → MP・詠唱・スロットのような魔法システムは不要。**魔法スキルを「イベント分岐を開く鍵」に
  する** — Healing 持ち同行で瀕死救助率が上がる、Arcane Shield 持ちは冒険 hazard を1段緩和する程度で
  十分。「治癒師がいたから生還した」が歴史に残ればもうフレーバーではない
- **種族** → 寿命比率の加齢帯化（§3）に加え、**種族→部族→言語 family→命名という縦糸が実装済み**
  なので文化との接続が最も自然。種族ごとのイベント重み・初対面の関係補正は数行で入る

いずれも戦闘ラウンド化・Personal Mode・言語知識という既定路線の中で解消され、独立した大工事は不要。

## 12. TRPG 的戦闘＋魔法の設計案（探索遥かなり2スタイル）

### 12.1 好材料

- `combat.py` は既に**ヘッドレス・副作用なし・`Combatant` Protocol・構造化 `CombatLogEntry`
  （to_dict 付き）**の正しい骨格。差し替えるのは `resolve_combat` の中身（1回比較→ラウンドループ）と
  `_build_combat_log`（後付け演出→判定の写し）のみ
- スキルカタログ（`world_data.py:37-71`）は 8 カテゴリ 54 種。Magic 11 種・Divine 7 種が既に
  「呪文の名前」をしている

### 12.2 判定式 — 2d6 + 修正 vs 目標値

```
攻撃ロール = 2d6 + 能力修正 + スキルLv
目標値     = 7 + 相手の回避修正（dex / Evasion / シールド系バフ）
マージン   → かすり / 命中 / 会心 の3段階
```

**ロールの生値・修正・目標値をすべてログに残す** — これが「嘘ログ」（結果からの逆算演出）の根絶であり、
探索遥かなり2 の読み味の正体。判定とログは同一乱数系列から生成する。

### 12.3 魔法 = スキル駆動の行動（魔法システムは作らない）

各ラウンドの行動候補を所持スキルから導出。スペルはデータテーブル1枚:

```python
SPELL_EFFECTS = {
    "Fireball":      {"kind": "attack", "stat": "intelligence", "power": 2},
    "Holy Light":    {"kind": "heal",   "power": 2},
    "Arcane Shield": {"kind": "shield", "duration": 2},
    "Battle Cry":    {"kind": "buff",   "target": "allies", "mod": +1},
    "Entangle":      {"kind": "debuff", "target": "enemy",  "mod": -2},
    ...
}
```

NPC の行動選択は軽量ポリシー（味方重傷ならヒーラーは回復優先等）。
**この行動選択がそのまま Personal Mode の差し込み口**（pending_choice と同じパターン）。

### 12.4 HP 問題 — 世界モデルに HP を足さない

戦闘中だけの **vitality プール**（例: `10 + constitution // 10`）を使い、終了時に残量を
injury ladder へ写像（70%超→無傷 / 30〜70%→injured / 30%未満→serious / 0→dying）。
戦闘内は TRPG 的攻防、戦闘外は従来の ladder——既存の回復・救助・瀕死解決・「即死なしで介入余地」
の設計原則が無傷で残る。**直接死なせず dying 止まり**にし、死は既存の瀕死解決に委ねる。

### 12.5 ログ形式と表示

`CombatLogEntry` を拡張（`action_kind` / `skill_key` / `dice` / `modifier` / `target_number` /
`effect`）し、表示は `summary_key` + `render_params` の遅延レンダリング（既定路線）。
文面バリエーションは `narrative/template_history.py` の冷却機構を流用。

```
R1  エラリア《ファイアボール》 2d6=9 +魔力4 → 13 vs 回避10 ◆命中（活力 -6）
R1  オルグ《斬撃》            2d6=7 +武勇3 → 10 vs 防御12 ◇外れ  シールドが刃を逸らす
── 決着: エラリアの勝利（3R）。オルグは深手を負い逃走（serious） ──
```

### 12.6 利用箇所は3つ・実装は1つ

1. 個人戦闘イベント（`events/combat.py`）— 差し替え
2. 冒険 hazard ステップ — 単発負傷率ロールを小規模戦闘に置換（スキル接続が同時達成）
3. 戦争アークの会戦 — N対N は分隊単位に丸めて同じ resolver（代表ロール＋規模係数）

### 12.7 移行・テスト

- `combat_log` payload 形式変更 → **スキーマ v9**（WorldArc と同梱可）
- 決定論テスト（同 seed→同ログ同結果）必須
- 統計ハーネスに勝率分布・戦闘死亡率・魔法使用率・平均ラウンド数を追加。
  負け犬スパイラル調整（敗者ペナルティ縮小）も同時に計測しながら行う

実装順: (1) ラウンド制化＋構造化ログ＋i18n テンプレ → (2) SPELL_EFFECTS と行動ポリシー →
(3) 冒険/戦争接続 → (4) Personal Mode 用行動選択シーム

## 13. 言語エンジン方向性の判定

**結論: docx 設計書（§7–8、LG-1〜6）の方向性で正しい。修正1点＋順序の注意2点。**

### 13.1 そのまま採用するもの

- 「意味→音」生成と生成時の由来保存（`ToponymEtymology` / `ToponymComponent`）。後付け推測は
  save/load 後に由来が揺れる（docx 自身のリスク指摘どおり）
- `LocationNameRecord`（現地名/公用名/旧名/通称の履歴）
- `SemanticRootDefinition` / `LanguageRootRealization`
- **表示先は既にある**: マップ詳細パネルに `generated_endonym` / `name_etymology_line` / `aliases` の
  表示枠が実装済み（`map_location_renderer.py:334-351`）。LG-1 はこの枠に流す中身を本物にする話

### 13.2 修正点: 「いつ・なぜ進化するか」が設計書に欠けている

docx は言語体系の**構造**は精密だが、進化トリガーは現状の `evolution_interval_years` タイマーのまま。
これは「因果のないサイコロ」（§5）と同じ穴。歴史駆動化——占領→借用語、ルート長期封鎖→方言分岐、
era シフト→命名流行、`LanguageEvolutionRecord` への `cause_event_id` 追加——を設計書に追補する。
docx の構造設計と矛盾せず合成可能。

### 13.3 発見: 着手前に繋ぐべき既存の断絶

world_change の地名変更ジェネレータ（`world_change_driver.py:101-118` `_rename_candidates`）は
**"Frontier Hold" / "X March" のような英語テンプレで名前を作っており、言語エンジンを使っていない**。
LG 計画の最初の一歩は「リネーム発生時、支配派閥/地域の言語で言語エンジンから新名を生成する」接続。
新規データモデル不要・既存機能同士の配線のみで、地図上の名前に言語の手触りが出る。

### 13.4 順序の注意

- **LG-4（seed から複数言語体系の自動生成）は後回し**。最重量かつ「worldgen PoC 未完了」
  （CLAUDE.md）の領域。手書きの Aurelian/Thornic 系で LG-1〜5 は全部成立する
- **LG-6（個人モード言語知識）は Personal Mode 依存**なのでその実装期に合わせる

推奨順: リネーム⇔言語エンジン接続 → LG-1 → LG-2 → LG-5 → LG-3 →（Personal 期に）LG-6 →
（worldgen 期に）LG-4。

**警告**: 言語は最も楽しい沼。各 LG ステップの完了条件を「プレイヤーの画面に見える成果物」で定義する
（例: LG-2 完了＝マップ詳細に「Kharum＝暗い峠（Old Thornic）」が出る）。

### 開発の合言葉（docx 付録B を採用）

- Observer では世界を**読む**。Sandbox では世界を**編集する**。Personal では世界に**生きる**。
  ただし、どのモードでも動いている世界は同じ
- 事件を増やすより、事件の**余波**を見せる
- 地名を後から説明しない。**意味から生成し、音に変換し、由来を保存し、UI で読ませる**
- （本レポート追加）変化はサイコロからではなく、**前の状態から因果的に**導く

## 15. World Health Check — 「世界が死んでいる」を検出するテスト設計

開発主体が AI エージェント（Claude Code / Codex）であることを前提にする。
**AI は「pytest が落ちる」「CLAUDE.md / AGENTS.md に書いてある」ことしか確実には発見できない**ため、
世界の病理をこの 2 チャネルへ流し込む。手動スクリプト（`_playtest_*.py` 方式）は回した人しか
発見できないので、常設の検出器にはならない。

### 15.1 原則: 「別スクリプト」ではなく「落ちる pytest」

ロードマップ §14 の 0b（統計ハーネス常設）の具体設計。docx P1-3 の統計テスト例
（「全員死亡しないか」「marriage が偏りすぎないか」等）は本レポートが実測で見つけた病理を
ピンポイントで検出する設計だった——pytest に常駐していれば §2 の問題は自動検出されていた。

### 15.2 2 層構成

**第 1 層: 高速スモーク**（通常 pytest に常駐、固定シード 1〜2 本 × 20〜30 年、数秒）
`tests/test_world_health.py` — 絶対に破れてはいけない**不変条件**:

- 生存キャラに、active リストに存在しない `active_adventure_id` を持つ者がいない（バグ #1 を 1 行で検出）
- 関係性・冒険・噂のダングリング参照ゼロ
- save→load→advance ＝ advance のみ、の決定論一致（RNG ラウンドトリップ）
- 月中 save/load 後に LocationState がベースラインへ収束（季節デルタバグ #3 の回帰）

**第 2 層: 統計ハーネス**（`@pytest.mark.simulation_stats`、20〜100 シード × 50 年、
quality_gate / 夜間 CI で実行）`tests/test_world_health_stats.py` — **ヘルスバンド**（許容帯）:

| 生命徴候 | バンド例 | 今回検出できたはずの病理 |
|---|---|---|
| 人口推移 | 50 年後に初期の 40% 以上 | 人口絶滅 |
| 結婚発生率 | 50 年 × 20 人で 1 件以上 | 結婚 0 件 |
| 冒険結果分布 | death 0.5〜10%、safe_return 80% 未満 | 悲劇ゼロ・分布平坦 |
| 関係値分布 | クランプ（±100）張り付き率 < 20% | 飽和・ドリフト |
| era シフト | ≤ 3 回 / 100 年 | 16 回 / 100 年 |
| 戦争の中身 | 戦争 1 件あたり関連イベント ≥ N | 空っぽの戦争 |
| エルダーの多様性 | 高齢キャラのステ分散 > 閾値 | wis100 / dex1 への全員収束 |
| イベント構成 | aging 比率 < 60% | 49%（境界ケース） |
| save サイズ | 50 年で < X MB | レコード異常増殖 |

### 15.3 AI エージェントが「発見し、正しく行動できる」ための 4 つの仕掛け

1. **自己完結型の失敗メッセージ**: 観測値・許容帯に加え「ゲーム的に何を意味するか」「次にどうすべきか」
   「参照ドキュメント」まで含める:

   ```
   AssertionError: [world-health] marriage_count=0 (expected >=1 per 50y/20chars, seeds=[7,42])
     意味: 結婚が成立せず世代交代が起きない（世界が一代で絶滅する）
     対応: (a) 意図的なバランス変更なら tests/support/balance_expectations.py の帯を同一PRで更新
           (b) 心当たりがなければ events/relationships.py の閾値/関係値ドリフトを調査
     参照: docs/review_report_2026-06-10.md §2
   ```

2. **バンドの一元化**: `tests/support/balance_expectations.py` に全許容帯＋理由コメント。意図的リバランス時は
   同一 PR でバンドを更新する運用 —「テストを黙らせる」のではなく「設計判断を記録する」場所にする
3. **既知の未修正問題は `xfail(strict=True)` で台帳化**: 例: 結婚 0 件は修正まで xfail。修正が入ると
   **XPASS で逆に落ちて「直ったので xfail を外せ」と AI に通知される**。アーキテクチャガードの
   `removal_condition` 付き override と同じ負債台帳思想で、既存文化と整合
4. **CLAUDE.md / AGENTS.md に運用節を追加**: ヘルスチェックの読み方・バンド更新の判断基準・
   stats マーカーの回し方。AI エージェントが確実に読む場所はここしかない

### 15.4 配置

- `fantasy_simulator/testing/world_metrics.py` — ヘッドレスなメトリクス収集器
  （`_playtest_*.py` のロジックを移植・正式化）。テストとアドホック調査の両方から import
- メトリクスを毎回 JSON でも出力（`.world_health_report.json` 等、gitignore 対象）—
  テストが**通っていても** AI が前回実行との差分を読める
- CI: 第 1 層は既存マトリクスに自動で乗る。第 2 層は `quality_gate.py strict` への組み込み
  または夜間ワークフロー
- 決定論: シードリスト固定、失敗時にシードを必ず出力（再現可能性）

## 16. UI ロングターム方針 — TUI → GUI → ゲームエンジンの判断

### 16.1 結論

- **TUI → GUI の順（docx UI ロードマップ）は、将来のゲームエンジン採用を検討していても変更不要**
- ゲームエンジンは「分岐」ではなく「同じ道の先」にできる。鍵は AppService 境界の設計規律 1 点のみ

### 16.2 なぜ TUI が先か（エンジン検討中でも変わらない理由）

Textual TUI ステップの本当の成果物は画面ではなく**情報設計**（何を見せるか・どんなコマンドがあるか・
観察ループの動線・噂板/年代記/地図詳細の構造）。これはどのフロントエンドでも必要で、TUI が最安の
検証環境。docx の「最初から GUI へ飛ぶと技術選定・画面設計・ゲーム設計が同時に重くなる」はエンジンには
倍適用される。今エンジンを始めるとシーン構築・アセット作業が情報設計の検証を圧迫し、世界はまだ
死んでいる（§2）ので映すものがない。

### 16.3 エンジン採用時の構成 — Python コアは捨てない

言語境界の解は 3 つあるが、実質 1 択:

1. シミュレーションを GDScript/C# に移植 — 7.3 万行＋同規模テストを捨てる。**論外**
2. エンジンに Python を埋め込む — プラグインが脆弱で保守リスク高。非推奨
3. **プロセス分離: Python シムをヘッドレスサーバとして残し、エンジンは JSON を話すフロントエンド** — 本命。
   ターン制の観察ゲームなので IPC レイテンシは問題にならない

### 16.4 今やっておく保険（コストほぼゼロ・必須）

AppService（docx UI-0）設計時に、**コマンドと ViewModel を JSON シリアライズ可能なプレーンデータ
（dataclass/dict）に限定し、生の World/Character オブジェクトを境界を越えて渡さない**。
Python 内の TUI/GUI なら生オブジェクトでも動いてしまうが、それをやるとエンジン移行時にプロセス境界が
引けなくなる。アーキテクチャガードに 1 ルール（`app/` 層の戻り値型制約）で機械的に強制可能。
この規律を守る限り、Godot・Tauri・Unity・ブラウザのどれもが「JSON を話すクライアント」として
同じバックエンドに挿さり、**エンジン選定は最後まで遅延できる**。

### 16.5 エンジン候補の評価（本プロジェクトの特性: Python コア活用・テキスト主役・2D・ターン制・ソロ＋AI 開発）

| 候補 | 強み | 弱み | 向くケース |
|---|---|---|---|
| **Godot**（第一候補） | 軽量・無料・2D 第一級。RichTextLabel(BBCode)。WebSocket/TCP で Python 分離が容易。Steam/itch 輸出が枯れている。ライセンスリスクなし | 高度なテキスト組版は HTML に劣る。GDScript の LLM 習熟は C#/Web よりやや下 | 「観て・生きる」ゲームに育てる。動く地図・演出・音まで視野 |
| **Web スタック**（Tauri + Python サイドカー / ローカル Web） | **テキスト描画の王者は HTML**。年代記・噂板・語源表示・検索は圧倒的に楽。AI エージェントの生成精度が全候補中最強。NiceGUI 路線と地続き | 「エンジン感」はない。リアルタイム演出・音は弱め。Python 同梱パッケージング（PyInstaller サイドカー）がやや面倒 | 「読む」ゲームの完成形を最高の組版で出す（DF Legends ビューアー的製品） |
| **Unity** | C# エコシステム・アセット最大。LLM 訓練データ最大級。コンソール/モバイル展開なら独壇場 | テキストシムには重装備。ソロ開発の固定費が高い。ランタイム課金騒動以降の信頼問題 | 将来コンソール展開・商業規模を本気で狙う場合のみ |

**消去した候補**: Unreal（ジャンル不一致）/ pygame・Arcade（UI 全自作になる。Python のままが
欲しいなら Web の方が楽）/ Ren'Py（VN の線形フロー専用でダッシュボード型 UI と相互作用モデルが
合わない）/ MonoGame・LÖVE・Bevy（UI 自作＋言語境界の二重苦で Godot への優位なし）/
RPG ツクール・ウディタ（Python コアを外部に持てない閉環境）。

### 16.6 判断時期と判断基準

**Textual Observer MVP と Personal プロトタイプが動いてから**決める。基準はプロダクトの快楽の所在:

- 「文章を読み耽る」側 → Web スタック（HTML の組版力が効く）
- 「地図と世界が動くのを眺める」側 → Godot（シーン/アニメーションが効く）

AI エージェント主体の開発体制でのタイブレーカーは「LLM がそのスタックのコードをどれだけ正確に
書けるか」: Web > Unity(C#) > Godot(GDScript)。ただし Godot も実用水準であり決定打ではない。
